C'est un terme qu'on entend beaucoup dans la communaute JavaScript - tous les frameworks recents font leurs pubs et le mettent en avant.

Et bien que, comme beaucoup de techno, ca puisse faire peur, ca reste un concept relativement abordable si on prend le temps de decortiquer le truc. On va voir ca ensemble.

## C'est quoi ?

Le Server Side Rendering (connu sous l'acronyme SSR) fait exactement ce qu'il suggere : il "rend" (ou fabrique) du HTML sur un serveur et envoie ce HTML au navigateur.

**Pour faire du SSR, il est necessaire d'avoir un serveur qui tourne quelque part, sans arret**:

<SSRRequestTimeline client:only />

Le serveur va litteralement, et a chaque fois, effecter des traitements de calcul pour generer du HTML qui sera ensuite affiche dans le navigateur.

## Et c'est tout ?

Techniquement, et si on reste stricte dans les termes, oui, c'est tout. C'est une technique qui existe en fait depuis des annees : PHP, JAVA et C# fonctionnent de cette maniere pour generer des pages web depuis plusieurs decennies maintenant.

Avec l'experience, on a compris que le SSR au sens strict, etait vraiment super puissant pour ameliorer l'experience de chargement de la premiere page : on parle d'optimiser **the initial page load**.

## C'est tout, mais y'a mieux

Cependant, les frameworks JavaScript actuels sont alles plus loin : ils utilisent le SSR pour optimiser l'affichage de la premiere page mais redonnent la main au navigateur pour rendre les pages Web interactives afin de fournir une experience optimale.

<AppStartTimeline client:only />

Sur cette timeline, de nouvelles frames ont ete ajoute:

- le telechargement des scripts JavaScript present dans le HTML generes par le serveur
- l'execution de ses scripts
- le demarrage de l'app JavaScript

Les deux premieres sont effectuees par le navigateur en lui meme, il effectue son travail en interpretant le code HTML et en respectant les balises qu'il rencontre : s'il tombe sur un `<script src>`, il va telecharger le fichier associe, et l'executer.

La troisieme frame est la plus interessante : pour rendre la page web interactive dans le navigateur, le framework en question doit "demarrer". Tous les frameworks ne demarrent pas de la meme maniere : certains passent par une phase appellee [hydration](/concepts/hydration).
