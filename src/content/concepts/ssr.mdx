C'est un terme qu'on entend beaucoup dans la communauté JavaScript - tous les frameworks récents font leurs pubs et le mettent en avant.

Et bien que, comme beaucoup de techno, ça puisse faire peur, ça reste un concept relativement abordable si on prend le temps de décortiquer le truc. On va voir ça ensemble.

## C'est quoi ?

Le Server Side Rendering (connu sous l'acronyme SSR) fait exactement ce qu'il suggère : il "rend" (ou fabrique) du HTML sur un serveur et envoie ce HTML au navigateur.

**Pour faire du SSR, il est nécessaire d'avoir un serveur qui tourne quelque part, sans arrêt**:

<SSRRequestTimeline client:only />

Le serveur va littéralement, et a chaque fois, effectuer des traitements de calcul pour générer du HTML qui sera ensuite affiche dans le navigateur.

## Et c'est tout ?

Techniquement, et si on reste stricte dans les termes, oui, c'est tout. C'est une technique qui existe en fait depuis des années : PHP, JAVA et C# fonctionnent de cette manière pour générer des pages web depuis plusieurs décennies maintenant.

Avec l'expérience, on a compris que le SSR au sens strict, était vraiment super puissant pour améliorer l'expérience de chargement de la première page : on parle d'optimiser the **initial page load.**

## C'est tout, mais y'a mieux

Cependant, les frameworks JavaScript actuels sont allés plus loin : ils utilisent le SSR pour optimiser l'affichage de la première page mais redonnent la main au navigateur pour rendre les pages Web interactives afin de fournir une expérience optimale.

<AppStartTimeline client:only />

Sur cette timeline, de nouvelles frames ont ete ajoutées:

- le téléchargement des scripts JavaScript présent dans le HTML généré par le serveur
- l'exécution de ces scripts
- le démarrage de l'app JavaScript

Les deux premières sont effectuées par le navigateur lui-même, il effectue son travail en interprétant le code HTML et en respectant les balises qu'il rencontre : s'il tombe sur un `<script src>`, il va télécharger le fichier associé, et l'exécuter.

Le troisième frame est la plus intéressante : pour rendre la page web interactive dans le navigateur, le framework en question doit “démarrer”. Tous les frameworks ne démarrent pas de la même manière : certains passent par une phase appelée [hydration](/concepts/hydration).
